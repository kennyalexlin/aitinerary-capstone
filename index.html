About

Planning and booking a trip can be tedious and time-consuming. Travelers often juggle multiple websites just to find and book flights, hotels, and activities—re-entering the same information over and over. The process is fragmented, repetitive, and prone to error, especially when coordinating preferences and timing across a full itinerary.

We set out to simplify this experience by reimagining what travel planning could look like if it felt more like talking to a smart, capable assistant—one that understands your intent, fills in the gaps, and handles the logistics.

aitinerary uses large language models (LLMs) to interpret open-ended travel requests and extract structured details like dates, destinations, and preferences. These are mapped into clean, validated formats that power downstream automation. Once an itinerary is finalized, the system hands it off to an automated agent that navigates real booking websites on the user's behalf—filling out forms, selecting options, and completing the booking process end-to-end.

How It Works

1. Conversational Input Engine

Each user message flows through a structured multi-step LLM pipeline designed for stateful, multi-turn interactions:

  • Entity extraction identifies structured fields like departure city, destination, dates, budget, cabin class, and preferences.
  • Context summarization condenses the full conversation history—both user and assistant messages—into a compact snapshot of intent and progress.
  • Assistant reply generation produces a context-aware response based on the latest extracted fields and conversational context.

We use predefined data models such as FlightInfo, UserInfo, and Preferences to define what variables are required for automation and to store extracted values. These models are incrementally updated across turns, enabling partial inputs to build into a complete travel plan over time. They also serve as contracts between system components, ensuring clean, validated JSON is passed downstream—eliminating ambiguity and making automated execution reliable.

To handle typical inconsistencies in LLM outputs, we implemented lightweight post-processing modules, such as:

  • Date correction, which updates ambiguous or outdated year values based on the current date.
  • Airport resolution, which maps city names to IATA codes using a local airport database and prioritizes large international hubs when multiple matches exist.

2. Automated Booking Agent

Once the travel plan is complete and validated, it is passed to our custom web automation agent built on the open-source browser-use framework. This agent simulates human-like navigation on real booking websites—filling in forms, selecting flights, and completing checkout steps.

To handle the challenges of dense, dynamic booking UIs, we extended the base framework with:

  • Custom functions to manage edge cases like pop-up overlays, dropdown menus, and manual handoffs for sensitive input (e.g., payment fields).
  • Task-based segmentation, splitting the booking flow into discrete subtasks (e.g., search, selection, checkout) to reduce cognitive overhead and improve model focus.
  • DOM parsing and heuristics to prioritize actionable elements—such as buttons labeled "Search" or "Continue"—and match form fields using metadata like labels, placeholders, and accessibility tags.

This enables reliable, scalable automation of bookings across real-world websites without relying on brittle, site-specific templates.

3. Persistent Data Layer

Our backend supports a robust data storage and personalization layer, including:

  • User profiles, storing onboarding information like name, passport number, and travel preferences.
  • Session memory, tying all assistant interactions and updates to a unique session ID.

This enables:

  • Pre-filled fields and reduced user input during booking
  • Conversation continuity, allowing users to pause and resume planning
  • Long-term personalization, using past interactions to improve future suggestions and tailor the assistant experience over time.

Performance

Evaluation of the system was performed on the input and browser use aspects separately. Although we ideally wanted to test the system end-to-end, time constraints prevented us from fully integrating both components.

Input Testing

We began with manual testing to assess system performance and data extraction fidelity, using a variety of LLMs. This was followed by automated testing using a second LLM as a simulated user, allowing us to evaluate the system at scale.

We created a ground truth dataset of 100 trips in JSON format and defined four distinct user personas:

  • Comprehensive: Provides all required information in one message.
  • Direct simple: Provides one piece of information at a time, when prompted.
  • Error prone: Enters information incorrectly, then self-corrects.
  • Ambiguous: Provides vague input and clarifies when asked.

The simulated user was fed one of these personas and a trip from the dataset. It engaged in a back-and-forth conversation with the system until either all required fields were extracted or the conversation reached the 30-message limit.

We compared the system’s extracted fields to the ground truth, evaluating metrics such as:

  • Dialogue completion rate
  • Overall success rate (all required fields extracted without error)
  • Slot-filling precision, recall, and F1-score

We tested 400 combinations of trips and personas. The system achieved an overall success rate of 91%.

Browser Use Testing

We tested the browser automation agent manually across curated trips. Ten unique trips were run through the system three times each:

  1. Once without DOM improvements
  2. Twice with DOM improvements

Logs were collected and analyzed after each run. The key metric was end-to-end success, defined as the agent completing all four tasks without user aid:

  • Flight details page navigation
  • Flight selection
  • Form filling for personal details
  • Progression through booking steps

We also tracked total token usage, time per run, and cost. Without DOM improvements, the system achieved 44% success. With DOM improvements, it reached 100%.

Two key limitations of our evaluation process were:

  • Cost: $0.62–$2.50 per run
  • Time: Up to 16 minutes per test
